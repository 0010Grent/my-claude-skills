# 文件1：`CLAUDE.md`

```markdown
# arxiv Paper Reporter — Claude Code Skill 主配置

## Skill 概述
本 Skill 帮助用户阅读、理解全英文 arxiv 论文，并生成结构化、面向汇报讲解的论文解读报告。

---

## 核心能力

### 能力1：论文深度理解
- 解析论文的算法机制、数据集、实证结果
- 将数学公式转化为直觉性解释
- 识别论文核心贡献与创新点

### 能力2：汇报报告生成
- 生成面向听众的结构化解读报告
- 支持多报告人分工协作格式
- 每个板块包含：攻读目标、精读章节、汇报讲解要点
- 时间控制：为每个板块标注建议讲解时长

### 能力3：章节精读
- 针对用户指定章节进行深度解析
- 输出算法流程、关键机制、实验逻辑

---

## 可用命令

| 命令 | 功能 | 示例 |
|------|------|------|
| `/read-paper` | 读取并理解论文 | `/read-paper https://arxiv.org/abs/xxxx` |
| `/explain-section` | 精读指定章节 | `/explain-section 3.4,3.5` |
| `/write-report` | 生成完整汇报报告 | `/write-report --speakers 3 --duration 15min` |
| `/assign-speakers` | 分配报告人板块 | `/assign-speakers` |

---

## 工作流规范

### 标准工作流（推荐）
1. 用户提供论文 → 执行 `/read-paper`
2. 用户确认理解框架 → 执行 `/assign-speakers`
3. 针对每个板块 → 执行 `/explain-section`
4. 生成完整报告 → 执行 `/write-report`

### 报告质量标准
- **可读性**：避免术语堆砌，每个概念给出白话解释
- **逻辑性**：板块之间有承接，听众能跟上思路
- **完整性**：算法 + 数据 + 实证 三维度覆盖
- **时间感**：每个讲解要点标注预计耗时

---

## 输出格式规范

每个报告板块必须包含以下结构：

```
报告人 [X]：[主题名称] ([时长])

攻读目标：[1-2句话描述该板块的核心理解目标]

精读章节：
● [章节号] [章节名]
● [章节号] [章节名]

汇报讲解板块：
● [讲解要点1]：[详细解读内容]
● [讲解要点2]：[详细解读内容]
● [讲解要点3]：[详细解读内容]

听众理解检查点：
□ [关键概念] — 听众应能用自己的话复述
□ [核心机制] — 听众应能描述运作逻辑
```

---

## 重要行为准则

1. **先理解，再报告**：在生成报告前，必须完整解析论文
2. **双层解释**：每个技术概念给出「白话版」+「技术版」两层解释
3. **保留原文引用**：关键术语保留英文原文，括号内给中文解释
4. **数学公式必须解释**：不允许直接复制公式，必须解释其物理/直觉含义
5. **实验数据要解读**：数字背后的意义比数字本身更重要
```

# 文件2：`.claude/commands/read-paper.md`

```markdown
# /read-paper 命令

## 命令描述
读取并深度理解一篇 arxiv 论文，输出结构化的论文理解报告。

## 使用方式
```
/read-paper <arxiv链接 或 PDF路径 或 论文文本>
```

## 执行逻辑

当用户调用此命令时，按以下步骤执行：

### Step 1：获取论文
- 如果是 arxiv 链接：调用 `tools/arxiv_fetcher.py` 获取论文内容
- 如果是 PDF 路径：调用 `tools/pdf_parser.py` 解析 PDF
- 如果是粘贴的文本：直接处理

### Step 2：论文结构扫描
提取并输出以下信息：

```
📄 论文基本信息
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
标题：[原文标题]
作者：[作者列表]
机构：[所属机构]
发表时间：[年月]
arxiv ID：[ID号]

🎯 核心贡献（3-5条）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. [贡献1]
2. [贡献2]
3. [贡献3]

📚 论文章节结构
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[自动生成章节目录，每章附一句话说明]

🔑 关键术语表
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[术语英文] (中文解释)：[简短定义]
```

### Step 3：四维深度解析

**维度1：问题定义**
- 论文要解决什么问题？
- 现有方法的局限是什么？
- 为什么这个问题重要？

**维度2：算法核心**
- 核心方法/模型是什么？
- 算法的关键创新点在哪里？
- 训练/推理流程是怎样的？
- 重要公式的直觉解释

**维度3：数据与实验**
- 使用了哪些数据集？
- 实验设计的逻辑是什么？
- Baseline 对比了哪些方法？
- 消融实验验证了什么？

**维度4：结果与意义**
- 主要实验结果是什么？
- 数字背后说明了什么？
- 论文结论的可信度如何？
- 局限性和未来工作是什么？

### Step 4：输出论文地图

生成一个可视化的论文逻辑框架（用文字ASCII图表示）：

```
[问题背景] → [现有局限] → [本文方案]
                                ↓
                    [算法模块1] + [算法模块2]
                                ↓
                    [实验验证] → [结论]
```

### Step 5：推荐精读章节

根据论文内容，推荐哪些章节值得深度精读，并说明理由。

## 输出文件
自动保存到 `outputs/[论文标题]/01_paper_understanding.md`
```

# 文件3：`.claude/commands/explain-section.md`

```markdown
# /explain-section 命令

## 命令描述
对论文的指定章节进行深度精读和解析。

## 使用方式
```
/explain-section <章节号> [--depth <basic|deep|expert>]
```

示例：
```
/explain-section 3.4,3.5
/explain-section 3.4 --depth deep
/explain-section 4 --depth basic
```

## 执行逻辑

### Step 1：定位章节
在已加载的论文中定位指定章节内容。

### Step 2：章节解析框架

对每个指定章节，输出以下结构：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📖 章节精读：[章节号] [章节标题]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【本章在干什么】
用2-3句话说明本章的核心任务和在全文中的位置。

【核心机制拆解】
将本章的主要机制/方法拆解为步骤：

Step 1：[步骤名称]
  → 做了什么：[描述]
  → 为什么这样做：[原理]
  → 关键细节：[注意点]

Step 2：[步骤名称]
  → 做了什么：[描述]
  → 为什么这样做：[原理]
  → 关键细节：[注意点]

【数学公式解释】
[如果本章有公式，逐一解释]

公式：[原始公式]
白话解释：[用日常语言解释这个公式在做什么]
直觉理解：[为什么要这样设计]
关键参数：[每个参数代表什么]

【算法流程图】（ASCII版本）
[用文字图表示算法流程]

【与其他章节的关联】
- 承接自：[哪个章节的哪个概念]
- 为后续：[为哪个章节铺垫了什么]

【三个关键理解点】
⭐ 理解点1：[读懂这章必须抓住的第一个点]
⭐ 理解点2：[读懂这章必须抓住的第二个点]  
⭐ 理解点3：[读懂这章必须抓住的第三个点]

【汇报时的讲解建议】
- 用什么类比来解释这个机制？
- 哪个图表最适合拿来讲？
- 听众最可能在哪里迷惑？如何预防？
```

### Step 3：生成讲解稿草稿

可选：为该章节生成一段口语化的讲解稿，模拟报告人的讲解方式。

## 输出文件
自动保存到 `outputs/[论文标题]/sections/[章节号]_explanation.md`
```

# 文件4：`.claude/commands/write-report.md`

```markdown
# /write-report 命令

## 命令描述
基于已理解的论文，生成完整的结构化汇报报告。

## 使用方式
```
/write-report [选项]
```

选项：
```
--speakers <数量>      报告人数量（默认3人）
--duration <时长>      每个板块时长，如 15min、10-20min
--sections <章节>      指定每个报告人负责的章节
--style <style>        报告风格：academic/casual/mixed（默认mixed）
--output <路径>        输出路径
```

示例：
```
/write-report --speakers 3 --duration 15min
/write-report --speakers 2 --sections "A:1,2,3;B:4,5,6"
```

## 执行逻辑

### Step 1：报告整体规划

在生成具体内容前，先输出报告规划供用户确认：

```
📋 汇报报告规划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

论文：[标题]
总时长：[预计总时长]
报告人数：[X人]

分工方案：
┌─────────────────────────────────────────────────┐
│ 报告人A │ [主题]         │ 章节X.X, X.X │ XX分钟 │
│ 报告人B │ [主题]         │ 章节X.X, X.X │ XX分钟 │
│ 报告人C │ [主题]         │ 章节X.X, X.X │ XX分钟 │
└─────────────────────────────────────────────────┘

逻辑衔接：
A → B → C 的讲解逻辑是：[描述讲解逻辑链条]

是否按此方案生成报告？(确认/调整分工)
```

### Step 2：生成完整报告

用户确认后，生成以下完整报告结构：

---

**报告头部**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
论文解读汇报报告

论文标题：[标题]
论文来源：[arxiv链接]
报告日期：[日期]
报告人：[A、B、C]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
